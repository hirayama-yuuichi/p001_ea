

# 整理した内容



## やりたいこと

- 開発用 EC2（Windows）を、毎日 JST 9,21,0時に自動で stop する
- mt5/vscode/claude/ssh をインストール 

- 対象アカウントは以下の AWS IAM Identity Center から入る
  - https://identitycenter.amazonaws.com/ssoins-77588ac0c8111922
  - プロファイル AdministratorAccess-042608219431:
  - リージョン: `ap-northeast-1`
- CloudFormation で構築し、インスタンス名・インスタンスID・停止時刻（JST）をパラメータ化する



- スタック名: `ec2-stop-<インスタンスID>` （例: `ec2-stop-i-0123456789abcdef0`）
  - インスタンス名を含めたい場合は、ASCII の識別子に変換した上で `ec2-stop-<ascii名>-<インスタンスID>` とする
- `InstanceName` パラメータ自体は日本語可（説明文・Description表示にのみ使用し、リソース名には使わない設計にした）

## 停止方式

Lambda を新たに書かずに、以下の構成で実現します。

- **EventBridge スケジュールルール**を1本作成し、cron式の時刻フィールドにカンマ区切りで複数時刻を指定（例: `cron(0 0,12,15 * * ? *)`）
- ルールのターゲットは AWS が提供する **SSM Automation ドキュメント `AWS-StopEC2Instance`**
  - EventBridge 用 IAM Role: `ssm:StartAutomationExecution` と、Automation実行ロールへの `iam:PassRole` のみ
  - SSM Automation 実行用 IAM Role: `ec2:StopInstances` / `ec2:DescribeInstances` 等
- 時刻は **UTC の cron式をパラメータに直打ち**する（JST→UTC変換ロジックはテンプレートに持たせない）。対応表:

  | JST | UTC |
  |-----|-----|
  | 9時 | 0時 |
  | 21時 | 12時 |
  | 0時 | 15時（前日）|

作成したテンプレート: [`cloudformation/ec2-stop-schedule.yaml`](./cloudformation/ec2-stop-schedule.yaml)

# CloudFormation 実行手順

## 0. 前提

- AWS CLI v2 がインストールされていること
- IAM Identity Center へのアクセス権があること

## 1. SSOログイン

```bash
aws configure sso
# SSO start URL: https://identitycenter.amazonaws.com/ssoins-77588ac0c8111922
# SSO region: ap-northeast-1
# プロファイル名: AdministratorAccess-042608219431
```

既にプロファイルがある場合はログインのみ:

```bash
aws sso login --profile AdministratorAccess-042608219431
```

## 2. パラメータの確認

- `InstanceId`: 対象の開発用EC2（Windows）のインスタンスID（`i-` で始まるもの）
- `InstanceName`: 任意の識別ラベル（日本語可、Description表示のみに使用）
- `Schedule`: UTC cron式を直打ち。省略時は `cron(0 0,12,15 * * ? *)`（JST 9,21,0時 相当）

## 3. テンプレートの検証（任意だが推奨）

```bash
aws cloudformation validate-template \
  --template-body file://1000_mt5シュミレータ/開発用ec2/cloudformation/ec2-stop-schedule.yaml \
  --region ap-northeast-1 \
  --profile AdministratorAccess-042608219431
```

## 4. デプロイ

`<インスタンスID>` は実際のIDに置き換える。スタック名はアンダースコアが使えないためハイフン区切り。

```bash
INSTANCE_NAME=mt5
INSTANCE_ID=i-06f228c7623eba3a7

aws cloudformation deploy \
  --template-file ec2-stop-schedule.yaml \
  --stack-name "ec2-stop-${INSTANCE_ID}" \
  --parameter-overrides \
      InstanceName="${INSTANCE_NAME}" \
      InstanceId="${INSTANCE_ID}" \
      Schedule="cron(0 0,12,15 * * ? *)" \
  --capabilities CAPABILITY_NAMED_IAM \
  --profile AdministratorAccess-042608219431
```

`CAPABILITY_NAMED_IAM` が必要な理由: テンプレート内でIAM Roleに明示的な名前 (`RoleName`) を付けているため。

## 5. 動作確認

- EventBridge コンソールで `ec2-stop-<InstanceId>` ルールが `ENABLED` になっていることを確認
- 次回の停止時刻を待って自動停止されるか確認するのが確実。すぐ確認したい場合は、EventBridge Rule自体は手動起動できないため、代わりにSSM Automationを直接実行してロール・権限の設定が正しいかを個別にテストする:

```bash
aws ssm start-automation-execution \
  --document-name AWS-StopEC2Instance \
  --parameters "InstanceId=${INSTANCE_ID}" \
  --region ap-northeast-1 \
  --profile AdministratorAccess-042608219431
```

## 6. 削除する場合

```bash
aws cloudformation delete-stack \
  --stack-name "ec2-stop-${INSTANCE_ID}" \
  --region ap-northeast-1 \
  --profile AdministratorAccess-042608219431
```

## 注意事項

- 停止対象がスポットインスタンスや自動起動設定と組み合わされている場合、意図せず再起動されるケースがあるため事前に確認すること
- このテンプレートは「停止」のみ。起動（start）が必要な場合は同様の構成で `AWS-StartEC2Instance` を使った別スタックを追加する
